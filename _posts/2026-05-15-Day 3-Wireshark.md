### Lab Overview

In this lab, I used Wireshark to monitor network traffic between my Kali Linux VM and Windows VM while performing basic network communication and reconnaissance activities.

### Main activities:

observing background traffic
ICMP ping testing
Nmap scanning
comparing Windows Public vs Private network behavior


1. Initial Wireshark Capture (Background Traffic)

![imagie](/assets/wireshark1.png)

Before generating any traffic manually, Wireshark was already capturing packets from the Windows VM.

Observed protocols:

DHCP
ARP
IGMPv3
ICMPv6

Important observations:

systems continuously generate background network traffic even when idle
ARP traffic was used for local IP-to-MAC address resolution
multicast traffic such as IGMP and ICMPv6 was automatically generated by Windows networking services

Example:

Who has 192.168.56.2?


2. Kali Linux Ping Test

![imagie](/assets/kaliping.png)


Results:

7 packets transmitted
7 packets received
0% packet loss

Important observations:

successful communication between Kali and Windows
low latency inside the Host-Only network


3. Wireshark Capture During Ping

![imagie](/assets/wireshark2 ping.png)

Wireshark captured:

ICMP Echo Request
ICMP Echo Reply
ARP traffic

Important observations:

Kali sent Echo Request packets to the Windows VM
Windows responded with Echo Reply packets
ARP resolution occurred before communication started

Example:

Echo (ping) request
Echo (ping) reply

This confirmed:

both VMs were successfully communicating through the internal virtual network


4. Initial Nmap Scan (Windows Network Profile = Public)

![imagie](/assets/wireshark3nmap.png)

Wireshark filter used:

tcp.flags.syn == 1

Important observations:

many SYN packets were generated by Nmap
reconnaissance activity was visible in Wireshark
most ports appeared filtered or ignored

Example scanned ports:

80
445
53
25

Analysis:

Windows Firewall restricted service visibility because the network category was configured as Public


5. Nmap Scan After Changing Windows Network Profile to Private

![imagie](/assets/kalinmap.png)

Command used:

nmap -sV 192.168.56.2

Detected open ports:

Port	Service
135	MSRPC
139	NetBIOS
445	SMB
5357	HTTPAPI

Important observations:

more services became visible after changing the network profile to Private
Windows trust settings directly affected exposed services and attack surface visibility


6. Wireshark Capture After Private Network Change

![imagie](/assets/wireshark4nmap2.png)

Wireshark showed a much more complete TCP communication process.

Observed activity:

SYN packets
SYN/ACK responses
ACK packets
HTTP requests
SMB-related traffic
RST/ACK packets

Important observations:

Nmap successfully established connections with several Windows services
service detection traffic became visible after the firewall restrictions were reduced

Example:

SYN → SYN/ACK → ACK

Additional traffic:

GET / HTTP/1.0
HTTP/1.1 503 Service Unavailable

RST/ACK explanation:

connection resets commonly generated during aggressive service detection and scanning activity


7. SYN Packet Filtering

![imagie](/assets/wireshark5nmap3.png)

Wireshark filter used:

tcp.flags.syn == 1

Purpose of the filter:

isolate TCP connection attempts generated by Nmap

Important observations:

SYN packets clearly showed which ports Nmap attempted to access
reconnaissance behavior became much easier to analyze visually

Example:

49623 → 445 [SYN]

This demonstrated:

how defenders can identify reconnaissance and port scanning activity using packet analysis tools like Wireshark
Final Observations

### Key takeaways:

Wireshark provides visibility into real network communication between systems
ICMP, ARP, TCP handshakes, and service detection traffic can all be observed directly
Windows firewall and network trust settings significantly affect scan results
Nmap reconnaissance activity is clearly identifiable through SYN packet patterns
packet analysis helps defenders understand how systems communicate internally and how scanning activity appears on a network
